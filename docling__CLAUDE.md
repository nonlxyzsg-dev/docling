---

## 1. Что это

Форк библиотеки [docling](https://github.com/docling-project/docling) — основного движка парсинга документов (PDF, DOCX, XLSX, PPTX, HTML) в markdown / JSON. Внутри: layout detection, table extraction, picture description, OCR-интеграция.

Используется через HTTP-сервис `docling-serve` (тоже наш форк, отдельный репозиторий: https://github.com/nonlxyzsg-dev/docling-serve).

---

## 2. Зачем форк

В upstream месяцами не движутся issue, критичные для нашего use case:
- #2285 — thread-safety `DocumentConverter` и `HybridChunker`
- #2635 — производительность VLM pipeline

Мы используем docling в продакшене корпоративного AI-кластера. Ждать upstream не можем. Параллельно ничего не мешает делать PR обратно — если фикс окажется generic.

---

## 3. Главная боль

При обработке документов с большим количеством картинок (типично 50–200 на документ) реальный параллелизм запросов к VLM-бэкенду намного ниже теоретически возможного. На документе 207 стр. / 84 картинки наш SGLang (28 параллельных слотов, очередь пустая всё время) обрабатывает запросы «волнами»: 60–100 секунд активной работы, затем 30–60 секунд простоя, затем снова волна. Итого 13 минут вместо ожидаемых 1.5–2 минут.

Подозрение: внутри docling формируются батчи запросов с синхронным барьером — все запросы в батче должны завершиться, прежде чем стартует следующий батч.

Цель форка — устранить эти барьеры, сделать поток запросов непрерывным.

---

## 4. Долгосрочное направление

В перспективе хотим, чтобы docling мог использовать наш **OCR SDK** (отдельный сервис на Qwen3.5-2B + PP-DocLayoutV3 с параллелизмом 32 воркера) как один из recognition backend'ов вместо только OpenAI-compatible VLM API.

Сейчас не реализовываем, но **архитектурные решения должны это учитывать**: не вшивай жёстких предположений «recognition = OpenAI VLM call». Любая абстракция должна быть готова к подключению нового backend.

## 5. Принципы работы

### 5.1. Минимально-инвазивные изменения
Каждый патч — будущий конфликт при rebase на upstream. Поэтому:
- Не переписывай файлы целиком, если можно точечно
- Группируй наши изменения в выделенные модули/хуки, а не размазывай по кодовой базе
- Каждый патч — отдельный коммит с префиксом `[fork]` в title
- Веди `CHANGELOG_FORK.md` в корне репозитория

### 5.2. Стратегия синхронизации с upstream
- Главная ветка: `main` (наша рабочая)
- Ветка `upstream-tracking` — еженедельный merge из upstream/main
- При rebase на новую версию upstream каждый `[fork]`-коммит пересматривается. Если соответствующий issue закрыт в upstream — наш патч удаляется.

### 5.3. Тесты обязательны
Любое изменение производительности — с бенчмарком на эталонном наборе документов (см. `benchmarks/` в `docling-serve`). Никаких «кажется быстрее».

### 5.4. Совместимость
- Публичный API библиотеки (что импортируют `docling-serve` и наш прокси) — не меняем без необходимости
- Имена параметров pipeline options — не меняем
- Поведение для документов, которые сейчас работают приемлемо — не ломаем

### 5.5. Когда сомневаешься — спрашивай
Это код в продакшене. Если решение неочевидно (выбор архитектуры, риск регрессии, выбор между подходами) — формулируй вопрос и жди ответа. Не угадывай.

### 5.6. Безопасность
- Никогда не вшивай реальные API-ключи / токены / пароли
- В примерах — плейсхолдеры (`<API_KEY>`)

## 6. Технический контекст инфраструктуры

### 6.1. Что вокруг docling
- **HTTP-сервис вокруг библиотеки:** `docling-serve` (наш форк, https://github.com/nonlxyzsg-dev/docling-serve). Главный entry point.
- **Прокси перед docling-serve:** `docling-proxy` (FastAPI), порт 5005. Маршрутизирует по типам PDF, инжектирует VLM-конфигурацию. Исходники: https://github.com/nonlxyzsg-dev/docling-proxy. Карта проблем upstream — там в коде и комментариях.
- **VLM backend:** SGLang v0.5.9 с моделью Qwen3.5-122B-agent-50k, OpenAI-compatible API через LiteLLM (порт 4000). 28 параллельных слотов, контекст 50K, без reasoning.
- **OCR SDK (отдельный сервис):** Qwen3.5-2B vLLM + PP-DocLayoutV3, 32 воркера, ~5 страниц/сек. Сейчас не используется docling, в перспективе — будет.

### 6.2. Типичная нагрузка
- PDF (тексты, сканы, смешанные), DOCX (включая legacy с OLE-формулами от MathType), XLSX, PPTX, HTML, .DOC из Confluence (MIME HTML)
- Размеры: от 1 до 300+ страниц
- Картинки: 0–200 на документ
- Языки: преимущественно русский
- Параллельная нагрузка: типично 1–3 документа одновременно, пиково до 10

### 6.3. Известные баги, которые сейчас обходим снаружи (в прокси или sed-патчем в Dockerfile)
1. **`docling_jobkit/orchestrators/local/worker.py:126`** — `SyntaxError: keyword argument repeated: exc_info`. Патчим sed'ом. *(это `docling_jobkit`, не сам `docling` — но связано)*
2. **Pillow `tile cannot extend outside image`** при `image_export_mode=embedded` + VLM pipeline. Прокси подменяет на `placeholder`.
3. **`do_picture_description=true` при VLM pipeline** инициализирует SmolVLM (`FileNotFoundError`). Прокси подавляет.
4. **`picture_description_custom_config` не уважает `concurrency`** — последовательная обработка картинок. Прокси переключает на `picture_description_api` (×2.2 ускорение, но барьеры между батчами всё равно остаются — это главная боль).

Найти источник этих багов в коде, оценить — фиксить ли нативно в форке.

## 7. Правила коммитов и общения

- **Общение со мной — на русском.** Всё: вопросы, комментарии, описания PR, ответы.
- **Commit messages — на русском.** Conventional commits в формате: `[fork] feat: краткое описание`, `[fork] fix: ...`, `[fork] perf: ...`, `[fork] docs: ...`, `[fork] refactor: ...`. Префикс `[fork]` обязателен.
- **Имена переменных, функций, классов, файлов — на английском** (это код).
- **Комментарии в коде — на английском** (для совместимости, если вернём в upstream PR).
- **Документация в `.md`-файлах — на русском.**

## 8. Workflow

- **Коммитим сразу в `main`** — feature branches не используем. Каждый коммит должен быть рабочим (тесты проходят).
- **`git push --force` на main — запрещено** без явного указания.
- **Плохие commit messages запрещены:** не `WIP`, не `fix`, не `update` — формулируй конкретно.

## 9. Структура задач

- **Каждая задача** = отдельный markdown в `TASKS/TASK_NN_<short>.md` (в репозитории `docling-serve`, потому что это главный entry point — задачи могут затрагивать оба репо, координатор там).
- **Каждая задача — две фазы:**
  1. **Анализ** — глубокое изучение, заключение с вариантами. Никакого кода.
  2. **Реализация** — после явного одобрения инженером — план + код + тесты + бенчмарки.
- **Между фазами — пауза для согласования.**
